「自分専用の秘書を作りませんか？」

まず、今日の研修が終わった先にある"未来の自分"の姿を、一緒にイメージしてみてください。

こんな状況、想像してみてください——

  • 「あの資料、どこいったっけ？」 → 秘書が探してきてくれる
  • 「先週の会議のメモ、要約しておいて」 → 秘書がまとめてくれる
  • 「このデータ、グラフにしといて」 → 秘書が作ってくれる
  • 「競合3社の最新動向、調べておいて」 → 秘書がリサーチしてくれる
  • 「このメール、丁寧な日本語に直して」 → 秘書が整えてくれる

「あれやっといて」「これってなんだっけ？」が、たった一つの窓口からすべて解決する。

しかも、ここがポイントなのですが——ただの検索ツールではありません。 調べるだけではなく、作業の実行までしてくれる"相棒"です。

🧱 一つ目の「わからない」：そもそも Claude Code って何？

22 / 379

Claude Code を使えるようにするっていっても、そもそも Claude Code って何?

現在 Anthropic 社が提供している「Claude」と名のつくサービスは、代表的なものだけで 4つあります（正確にはもっとありますが、違いが分かりにくいものを今回は代表的に4つ挙げます）。

  1. Claude（ブラウザ／デスクトップアプリのチャット版）
  2. Claude Cowork
  3. Claude Code（CLI版）
  4. Claudeアプリの Code モード（デスクトップアプリ内のCode機能）

まずはこの4つの違いを理解しましょう。

23 / 379

1️⃣ Claude（チャット版）

Claude はブラウザ、もしくはデスクトップアプリで使用できるチャットモードがメインのものです。ChatGPTやGeminiをイメージしてもらうといいと思います。

特徴としては、チャットをメインとして、ブラウザやデスクトップアプリを通してインターネット上でAIが動いてくれるものです。ですので、インターネット上にあるものとしか連携できません。

例えば代表的なものでいうと、Google系のツール（Gmail / Drive / Calendar など）との連携はできますが、自分のPC上にあるファイル、例えばダウンロードフォルダやデスクトップにあるファイルについてはアクセスできません。なので「今私のデスクトップには何のファイルがある？」と聞いても、理解できないんです。

24 / 379

🌥 ここで重要な概念：「クラウド」と「ローカル」

このインターネット上と自分のPC上を対比して、クラウドとローカルという言葉で表現します。

  • クラウド = インターネット上にあるもの
  • ローカル = インターネットにつながっていない、自分のPC上でだけ使えるもの

この二つの言葉は今後もよく出てきますので、「クラウド＝ネット上、ローカル＝手元のPC」ととらえておいてください。

25 / 379

2️⃣ Claude Cowork

Claude Cowork は、デスクトップアプリとして動く"作業エージェント"です。非エンジニアの方向けに、ファイル管理やタスク自動化をしてくれるツールで、あなたのPC上のファイル（=ローカル）を直接読み書きできるのが特徴です。

先ほどの Claude（チャット版）がクラウド上でしか動けなかったのに対し、Cowork は ローカルのファイル・フォルダに触れるのが大きな違いです。「デスクトップにあるこのPDFを整理して」「ダウンロードフォルダのCSVをまとめてExcelにして」といった、手元のファイルを触る仕事を任せられます。

ただし、注意点が一つ。この後に説明する Claude Code とできることはかなり近いのですが、トークンの消費量がCoworkのほうが多い傾向があります。つまり、同じ作業をさせても、Cowork のほうがサブスクの使用枠を早く使い切ってしまうということです。なぜかというと、Cowork は非エンジニアでも使いやすいようにGUI（ボタンや画面）で包んでいる分、裏側で追加のやりとりが発生しているためです。

→ エンジニア的な作業をするなら、後述の Claude Code のほうがコスパは良い、と覚えておいてください。

26 / 379

3️⃣ Claude Code（CLI版） ← 今日使うのはコレ

Claude Code は、ターミナル（黒い画面）の上で動くコーディングエージェントです。

ここで、Cowork と Claude Code の大きな違いが出てきます。それが GUI と CUI の違いです。

  • GUI (Graphical User Interface): アイコン・ボタン・ウィンドウで操作するタイプ。マウスで触れる。Finder や Cowork がこれ。
  • CUI (Character User Interface): 文字だけで操作するタイプ。黒い画面に文字を打ち込む。ターミナルや Claude Code がこれ。（CLI とほぼ同じ意味で使います）

「それなら、見た目わかりやすいGUI（Cowork やデスクトップ版のCodeモード）でいいじゃん」と思うかもしれません。ですが、ここにも差があります。

CLI版 Claude Code のメリット

  • トークン消費が少ない — 同じ作業をしても、CLI版のほうが燃費が良い
  • 動作が軽い — 余計な画面描画がない分、反応が速い
  • カスタマイズ性が高い — 設定ファイル、スラッシュコマンド、Hooks など細かく制御できる
  • サブエージェントや並列実行など、上級機能はCLI版のほうが強い
  • スクリプトに組み込める — 自動化しやすい

デスクトップ版Codeモード / Cowork のデメリット

  • トークンを食いやすい（先述の通り）
  • CLI版でできる高度な機能の一部が使えない／使いにくい
  • 細かい制御が効かず、見た目がリッチな分"重い"
27 / 379

結論として、「デスクトップ版のCodeモードを使ってもいいけど、本気で使うならCLI版のほうが得」と思ってください。

🛣 PATH（パス）

ここは少し難しいですが、超重要です。

PATH は、PCが「コマンドを探しにいく場所のリスト」です。

例え話：会社の内線電話をイメージしてください。「営業部の田中さんに電話して」と言えば、内線番号を知らなくても交換手が繋いでくれる。これは交換手が「田中さんの居場所リスト（=PATH）」を持っているから。PATH が通っていないと、「田中さんの内線番号は？」と毎回フルで指定しないと繋がらない状態になります。

🌿 そしてもう一つの超重要な理由：「ブランチ」で並列作業を可能にする

並列作業で起きる問題

思い出してください。オープニングでお話しした「秘書が4〜5人、並行で動く世界」。これ、実はそのままやろうとすると大事故が起きます。

ブランチ = 作業場所を枝分かれさせる仕組み

元のフォルダをコピーせずに、別の並行世界を作るようなイメージ。それぞれのエージェントに別々のブランチ（並行世界）で作業してもらえば、お互いの作業がぶつからない。

非エンジニアにとっての現実

Claude Code は、Bashコマンドを実行できます。つまり、あなたのPCに対してほぼ何でもできる状態です。

これが Hook の役割

Hook は、Claude Code に「自動ブレーキ」を付ける仕組みです。

具体的には、Claude が何か操作をしようとするたびに、実行の直前に自動的にチェックが入るような設定ができます。

  • 危険なコマンドを実行しようとしたら → 自動でブロック
  • 秘密情報のファイルを読もうとしたら → 自動でブロック
  • クラウドに強制プッシュしようとしたら → 自動でブロック

あなたが判断する必要はありません。事前に設定した"ブレーキ"が、自動で働いてくれる。

⚠️ Hook を設定しないとどうなるか

Hook なしで Claude Code を使い続けると、こんなリスクがあります。

リスク1：うっかり承認で大事故

Claude は操作の前に「実行していいですか？」と確認してきます。でも、業務で疲れていたり、急いでいたりすると、中身を読まずに「はい」を押しがち。そこに削除コマンドが混ざっていたら、一瞬で大事な資料が消えます。

リスク2：権限のゆるい承認の積み重ね

「このコマンドは毎回聞かれて面倒だから、常に許可」にすると、似た見た目の危険なコマンドも通ってしまう。rm file.txt を許可したつもりが、rm -rf ~/ まで通る状態になっていた、というトラップが実際に公式の事例として報告されています。

リスク3：秘密情報の意図せぬ送信

.env ファイル（API キーやパスワードが入っている）を Claude が読み込むと、その内容は Anthropic のサーバーに送信されます。業務情報が意図せず外部に出る可能性があるのです。

リスク4：クラウドバックアップの破壊

将来 GitHub と連携したとき、git push --force でクラウドの履歴を上書きされると、チームの作業が全部吹き飛ぶ可能性があります。

Hook を設定するとどうなるか

  • ✅ 危険なコマンドは人間の判断を待たずに自動ブロック
  • ✅ 秘密情報ファイルへのアクセスは構造的に防がれる
  • ✅ 「間違えて承認」が起きても、Hook で救われる
  • ✅ 逆に、安全なコマンドは確認なしでスムーズに通る → 使いやすさを損なわない

これが「使いやすさを残しつつ、セキュリティを確保する」バランスの正体です。

115 / 379

🎯 重要：バランスの考え方

ここでよくある誤解は、「厳しくすればするほど安全」という発想です。

実はこれ、逆効果なんです。

  • ガチガチに設定しすぎる → 毎回「確認」ダイアログが出て、結局"はい連打"になる → 意味がなくなる
  • 許可しすぎる → 危険な操作も通ってしまう → 事故の元

狙うべきは:

  • 明らかに危険なものだけ確実にブロック（= 絶対にやらせたくないこと）
  • 明らかに安全なものは自動承認（= 毎回聞かれるとウザい日常操作）
  • グレーゾーンだけ人間に確認（= 判断が必要なもの）

このバランスで設定すれば、「安全 × 快適」が両立します。

💡 例え話：秘書のチームにも役職がある

  • Opus = 部長クラス：重要な判断、全体戦略を任せる。時給は高いが、そこに見合う思考力がある
  • Sonnet = 実務リーダー：日常業務の8割はこの人。機動力とコスパのバランスが最高
  • Haiku = 新人アルバイト：定型作業を任せる。速くて安い

重要な原則:

計画は Opus、実行は Sonnet

深い思考が必要な局面（計画・設計・レビュー）ではOpus。 手を動かす段階になったらSonnetに切り替える。

なぜか？

  • Opusを常用すると財布が死にます（サブスクの使用枠もすぐ尽きる）
  • Sonnetで計画を立てると、重要な論点を見落とすことがある
  • 使い分けが、品質とコストを両立する鍵

🆕 新しいキーワード：/clear と /exit

Claude Code には、机を片付ける・作業環境を更新するための2つのコマンドがあります。 この2つは似ているようで、役割が全く違います。正しく使い分けるのが非常に重要です。

/clear — 会話履歴をリセット（机の上を片付ける）

  • セッションは続行したまま、会話の履歴を消す
  • CLAUDE.md は再読み込みされるので、ルール・記憶は保たれる

  • タスクを切り替える時、文脈が散らかってきた時に使う

/exit — セッションを完全終了（出社し直す）

  • Claude Code のセッション自体を完全に終了する
  • 次に claude と打って起動したら、完全にフレッシュな環境で再開
  • 重要：ここで初めて、設定ファイルの変更が反映される
  • .claude/settings.json の変更
  • MCPサーバーの追加・変更
  • Hookの追加・変更
  • ~/.claude/CLAUDE.md（ユーザー共通ルール）の変更
  • 一旦作業を区切るとき、設定を変えた後に変更を反映させたい時に使う

🆕 新しいキーワード：Claude API

ここで重要な前提を整理します。

実は、「API」という言葉は2種類の全く違うものを指して使われるので、混同しないでください。

① 一般的な API（= 各サービスが提供している"窓口"）

  • 例: Google Drive API、Gmail API、Slack API、Notion API など
  • 各サービスが「外部のプログラムから自分のデータを使えるように」提供している窓口
  • 基本的に無料（各サービスの無料枠内で使える）
  • 前章の MCPも、裏ではこの"一般的なAPI"を呼び出している

② Claude API（= Anthropic が提供する"AIの脳みそ"）

  • Anthropic（Claudeを作っている会社）が提供する、AIの思考能力を外部から借りるサービス
  • プログラムから直接 Claude（AI）に問い合わせできる
  • 使った分だけ料金が発生（従量課金）
  • 今ここで話しているのは、この Claude API のこと

Tips A：Slack Bot 作成の落とし穴

335 / 379

⚠️ 落とし穴①：トークンの種類を理解する

Slack には3種類のトークンがあり、それぞれ出来ることが違います。

トークン種類 プレフィックス 役割 制約
Bot Token xoxb- アプリの独立アイデンティティ 招待されたチャンネルしか読めない。スレッド取得は DM/MPIM のみ
User Token xoxp- ユーザーとして動作 プライベートチャンネルやスレッドも招待不要でアクセス可能
App-Level Token xapp- Socket Mode 専用 メッセージ送受信には使えない（接続用）
336 / 379

💡 どう使い分けるか

  • 基本は Bot Token（xoxb-）
  • アプリの独立ID、安定している
  • ユーザーが退職してもトークンは生き続ける

  • メッセージ送信、イベント受信など、ほとんどの用途をカバー

  • どうしても必要な場合のみ User Token（xoxp-）

  • 特定のユーザーの権限で動く必要がある場合
  • プライベートチャンネルにBotを招待できない場合

  • ただし ユーザーが退職するとトークンが使えなくなるリスクあり

  • Socket Mode を使う時だけ App-Level Token（xapp-）

  • 後述の理由で Socket Mode は推奨しない場面が多い
337 / 379

⚠️ 落とし穴②：スレッドは明示的に取得しないと読めない

Slack のメッセージ取得は、デフォルトでトップレベル（親メッセージ）しか返さない。 スレッドの返信は別途明示的に取得する必要があります。

具体的にどういうことか

例えば「#general チャンネルの今週の会話を全部読んで要約して」と頼んだ場合:

  • ✅ トップレベルの投稿は取れる
  • ❌ スレッドの返信は含まれない

議論の本筋はスレッドで進むことが多いので、これに気づかないと"重要な情報を取り逃がす"。

対処法

Bot に依頼するときは、明示的にスレッドも含めるよう指示する:

#general チャンネルの今週の投稿を読み込んでほしい。 トップレベルのメッセージだけでなく、スレッドの返信も全部含めて要約してほしい。 （内部的には、各メッセージに対して conversations.replies を呼んで、thread_ts があるメッセージは全て展開してください）

さらに注意

Bot Token（xoxb-）では、パブリック/プライベートチャンネルのスレッド取得に制約があります。 完全にスレッドを取り込みたいなら User Token（xoxp-）が必要になるケースがあります。

338 / 379

⚠️ 落とし穴③：プライベートチャンネルは明示的に指定が必要

Slack のデフォルト設定では、Bot はプライベートチャンネルに自動参加しません。

具体的に何が起きるか

  • 「全てのチャンネルから情報を集めて」と頼むと → パブリックチャンネルしか拾えない
  • プライベートチャンネルの内容は見えないまま、静かにスルーされる

対処法

  1. Bot を明示的にプライベートチャンネルに招待する - 各プライベートチャンネルで /invite @your-bot を実行

  2. 必要なスコープを付ける - パブリック: channels:history, channels:read - プライベート: groups:history, groups:read（←これを忘れがち） - DM: im:history, im:read - グループDM: mpim:history, mpim:read

  3. Bot への依頼時にプライベートチャンネルも明示

#secret-project というプライベートチャンネルがあります。 Bot はすでに招待済みで、groups:history スコープも持っています。 このチャンネルの過去1週間の投稿も含めて要約してください。

339 / 379

⚠️ 落とし穴④：Socket Mode は本番で使わない

Slack の公式ドキュメントで「本番では HTTP を推奨、Socket Mode は開発用途」と明記されています。 しかし多くのチュートリアルが Socket Mode を勧めるため、誤解しやすいポイントです。

Socket Mode の問題点

  • ⚠️ 公式が本番推奨していない
  • ⚠️ Slack Marketplace に公開するには HTTP 必須（Socket Mode は不可）
  • ⚠️ メッセージイベントが届かない/大幅に遅延する報告が継続的にある（公式Issueで報告多数）
  • ⚠️ 数時間ごとに WebSocket 接続がリフレッシュされる
  • ⚠️ 複数インスタンスでイベントが1つにしか届かない（ブロードキャスト不可）
  • ⚠️ メモリリーク報告もある

✅ 推奨：定期実行なら"ポーリング"が信頼性高い

定期的にSlackの情報を取得したい場合（例: 毎朝7時に前日の議論を要約）は、Socket Modeではなくポーリングを推奨します。

ポーリングとは:

  • 定期的に（5分ごと、1時間ごとなど）API を叩いてメッセージを取得する
  • HTTP リクエストなのでstateless で安定
