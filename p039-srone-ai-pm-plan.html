1. 背景・課題

SR-ONEグループ全体でAI開発案件が急増し、以下の問題が発生している：

課題詳細
全体像が見えないどのAI PJが進行中か誰も把握できていない
優先度が不明何から手をつけるべきか意思決定の基盤がない
リリース後の管理がない稼働中エージェントの監視・改善が属人的
アイデアが散逸Slackの各チャンネルにアイデアが散らばっている

2. 現状のインフラ

資産状態備考
Slackアイデア投稿ch稼働中投稿 → Notion自動追加が動作中
Slack進捗報告chBot未接続アイデアchとは別チャンネル
AI自動化アイデアボード（Notion）稼働中25件+蓄積。Slack連携で自動登録中
Slack → Notion自動登録要確認管理者・実装方式は未確認

3. 藤田さん要件

① アイディアバックログ（Idea Backlog）
フラッシュアイディアや未検討のものをすべて入れるDB

② プロダクトバックログ（Product Backlog）
開発対象を管理するDB。Priority / Owner / Due 必須。Priority順で運用

③ ラン / メンテナンスバックログ（Run / Maintenance Backlog）
リリース済みエージェントの運用タスク（監視・改善・不具合対応）

運用ルール: アイディアは①に追加 → 開発するものだけ②へ移動 → リリース後は③で管理

ゴール: Notion上に3DB構成のチケット管理基盤を構築し、AI開発の全ライフサイクル（アイデア → 開発 → 運用）を一元管理する。加えて、Slack⇔Claude Code⇔Notionの連携でメンバーの運用負荷を最小化する。

4. 提案: 3DB設計

4.1 ① アイディアバックログ（既存DB改修）

方針: 既存の「AI自動化アイデアボード」（25件+蓄積済み）を改修して流用。データ移行不要。

現状スキーマ: タイトル / ユーザー / カテゴリ / ステータス(アイデア→検討中→開発中→完了) / 概要 / 実装メモ / AI整理済 / 原文 / 登録日

改修内容

変更プロパティ型説明
追加起案者Person誰のアイデアか（既存「ユーザー」テキスト型はそのまま残す）
追加インパクトSelect大 / 中 / 小 — トリアージ時の判断材料
追加実現難易度Select低 / 中 / 高 — 技術的な難しさ
追加→ プロダクトBLRelation②に昇格した場合のリンク
変更ステータスStatusアイデア → トリアージ待ち → 採用（②へ） / 保留 / 却下
維持ユーザー, カテゴリ, 概要, 原文, 登録日, 実装メモ—そのまま活用
廃止AI整理済—ステータスで管理するため不要

ビュー構成

  • 全件一覧: テーブル、登録日降順
  • トリアージ待ち: 「トリアージ待ち」ステータスのみ表示（定例で使用）
  • インパクト×難易度マトリクス: ボード形式、インパクト別グループ

4.2 ② プロダクトバックログ（新規DB）

目的: 「開発する」と決まったものだけを管理。ここにあるものだけを開発する。

チケット粒度ガイドライン:
  • 1チケット = 独立してリリースできる最小のデリバリー単位
  • 大きなPJはエピック（親チケット）+ 個別チケットに分解
  • 目安: 1チケット = 1〜2週間で完了できる粒度。1ヶ月超は分割すべきサイン
プロパティ型必須説明
タイトルTitleYesPJ / 機能名
概要TextYes何を作るか・なぜ作るか
PrioritySelectYesP0（緊急）/ P1（高）/ P2（中）/ P3（低）
OwnerPersonYes開発担当者
DueDateYes完了期限
ステータスStatusYesBacklog → Ready → In Progress → Review → Done → 中止
カテゴリMulti-Select—社内ツール / 顧客向け / インフラ / 分析 等
サイズSelect—S / M / L / XL（工数の目安）
スプリントSelect—Sprint 1, 2, 3...（運用に慣れてから導入可）
完了条件Text—何をもって完了とするか
技術メモText—実装方針・アーキテクチャメモ
← アイディアBLRelation—①からのリンク（出自の追跡）
→ ラン/メンテBLRelation—③へのリンク（リリース後の追跡）

ビュー構成

  • Priority順: テーブル、Priority昇順 → Due昇順（メインビュー）
  • ボード（ステータス別）: カンバン形式で進捗を可視化
  • Owner別: グループ化で誰が何を持っているか一目瞭然
  • 今週のスプリント: スプリントフィルタ（将来用）

4.3 ③ ラン / メンテナンスバックログ（新規DB）

目的: リリース済みエージェント・ツールの運用タスクを管理。

プロパティ型必須説明
タイトルTitleYesタスク名
エージェント名SelectYes対象のエージェント / ツール名
種別SelectYes監視 / 改善 / 不具合 / その他
ステータスStatusYesOpen → 対応中 → 完了
OwnerPersonYes対応担当者
DueDate—対応期限
優先度Select—高 / 中 / 低
概要Text—詳細説明
対応ログText—対応内容の記録
← プロダクトBLRelation—②からのリンク（元の開発チケット）
運用ルール: 新エージェントを②からリリースして③にエントリ作成する際、③の「エージェント名」Selectに選択肢がなければその場で追加する。

定例トリアージ: 固定参加は藤田さん + 堀江の最小2名。他の関係者は非同期（Notionコメント or Slack）で参加。

5.2 例外フロー

ケース対応
②のチケットが技術的に不可能と判明ステータスを「中止」に変更。①のリンク先を「保留」に戻す。定例で共有
③の不具合が大規模改修を要する②に新チケットを直接起票（①を経由しない）。③チケットに②へのリンクを追記
緊急依頼（P0）②に直接起票OK（①を経由しない）。ただし定例で事後報告
リリース前に仕様変更②の既存チケットを更新（新チケット不要）

5.3 定例トリアージのアジェンダ案（30分）

  1. ①の新規アイデア確認（5分）: 新着をざっと確認、不明点は起案者に質問
  2. 採用/却下判定（10分）: インパクト×難易度で判断、採用なら②に移動
  3. ②の進捗確認（10分）: ステータス更新、ブロッカー確認
  4. ③の運用状況（5分）: 不具合・改善要望の確認
初期トリアージの進め方: 既存25件+を一度にやらない。初回は「既に開発中/完了済み」を②③に振り分けるのみ。未検討アイデアは5件/回ずつ段階的にトリアージする。

5.4 進捗更新の動機設計

  • 進捗レポートBotに「未更新チケット」を表示: 1週間以上ステータス変更がないチケットをOwner名付きで表示
  • 低摩擦な更新手段: Claude Codeスキルからワンコマンドでステータス変更可能に
  • 定例でのステータス確認: ②のOwner全員の進捗を口頭確認

8. 判断ポイント

本計画を進めるにあたり、以下の判断が必要です。ご確認をお願いします。

  1. スキーマ承認: セクション4の3DB設計でOKか？追加・変更したいプロパティはあるか？
  2. チケット粒度: 「1チケット = 独立してリリースできる最小単位」でよいか？エピック概念は必要か？
  3. 定例頻度: 週次でよいか？隔週の方がよいか？
  4. 定例参加者: 固定は藤田+堀江の2名、他は非同期参加でよいか？
  5. Priority体系: P0-P3の4段階でよいか？MoSCoW（Must/Should/Could/Won't）の方がよいか？
  6. Phase 1aのスコープ: DB構築のみ先行でよいか？Slack連携を待ちたいか？
  7. 初期のスキル利用者: 堀江が代行操作する前提でOKか？SRAメンバーに最初から使ってもらいたいか？
