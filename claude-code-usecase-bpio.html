コンセプト: Gitリポジトリ = 秘書のデスク

1つのGitリポジトリに、業務データ・タスク・プロジェクト・ナレッジ・連絡先・意思決定記録を全て格納。Claude Code がこのリポジトリを「デスク」として、全ての業務を遂行する。

従来: メーラー → Slack → 会計ソフト → スプレッドシート → ブラウザ → エディタ を行き来
現在: Claude Code のターミナル1つで全てが完結

なぜ一元化するのか: コンテキストスイッチをゼロにし、全ての操作がGit履歴に残り、繰り返し作業がスキル化され、失敗が学習資産になる。

リポジトリ構造 = 業務の全履歴

1つのリポジトリが「秘書のデスク」として機能する。全てのデータはYAML frontmatter付きMarkdownで管理され、Git履歴で完全追跡される。

data/ logs/ ← 日次の全活動ログ（何をいつやったか） tasks/ ← TODO管理（active/ → completed/ に移動） projects/ ← 35プロジェクト（各々に専任エージェント設定） contacts/ ← 顧客・関係者DB（16名） decisions/ ← 意思決定の記録（背景・選択肢・理由） inbox/ ← Gmail/Slack等からの受信データ knowledge/ ← 調査・ナレッジ蓄積 notes/ ← 議事録・メモ .meta/ ← マスターテーブル（projects.json等） .state/ ← セッション間の状態引き継ぎ

プロジェクト = 独立したエージェント

各プロジェクトは独自のディレクトリを持ち、AGENT.md で専任エージェントの振る舞いを定義する。/project p006 と呼べば、そのプロジェクトの文脈を全て読み込んだ状態で応答する。

data/projects/active/p006-senjin-hr/ AGENT.md ← 専任エージェントの設定・振る舞い overview.md ← プロジェクト概要 contacts.md ← 関係者一覧 tasks/ ← PJ固有タスク docs/ ← PJドキュメント群

コミットメッセージは種別プレフィックスで分類。git log が業務日誌になる。

Hook = 安全装置とガードレール

自律的に動く代わりに、Hookで事前・事後のチェックを自動実行する。「やっていいこと」と「やってはいけないこと」をシステムレベルで制御。

タイミングHookやること
PreToolUse Notion書き込み制御 許可された3DBスコープ以外への書き込みをブロック
PreToolUse 危険コマンド防止 rm -rf, git push --force 等を事前検知
PostToolUse 自動ログ ツール実行を自動記録
Stop 学習キャプチャ セッション中のエラー・発見をLesson候補として蓄積
StopFailure APIエラーログ 失敗時の詳細を自動記録
TaskCreated タスク構造検証 新規タスクのフォーマット・必須フィールドを自動チェック
例: 誤ってNotionの本番DBに書き込もうとすると → Hookが自動ブロック → 「このDBへの書き込みは許可されていません」とエラー表示。事故を未然に防止。

自己進化 = 失敗が資産になる仕組み

Claude Codeの操作中に発生した失敗・発見・パターンを蓄積し、システム自体を改善していく3層の学習ループ。

具体例

きっかけLesson昇格先
Notionの本番DBを誤って更新 「書き込みは許可スコープを限定すべき」 Hook: Notion書き込み制御（exit 2でブロック）
APIレートリミットで12バッチ中10件失敗 「並列数を制御すべき」 エージェント設定: maxTurns制限を追加
勝手に次ステップに進んで手戻り 「全ステップ間でユーザー確認を必須化」 SKILL.md: 確認必須ルール追加
信頼度スコア: 各Lessonは初期0.5 → レビューで+0.2 → 0.7以上で昇格候補。週次レビュー（/weekly）で集計・昇格提案される。
