Chapter 1: ToBカスタマーサクセスとは

ゴール: CS担当者の仕事の全体像をつかむ
CS担当者の役割定義: CS（カスタマーサクセス）担当者は「作業をこなす人」ではなく、「顧客の目標達成を支援する人」。採用BPOにおけるCSは、スカウト・日程調整・報告を通じて、顧客が採用目標を達成できる状態を維持し続けることが仕事です。

CS・サポート・営業・PMの違い

役割主な目的動き方成果の定義
CS（カスタマーサクセス） 顧客の継続的な成功支援 プロアクティブ（先回り） 顧客が目標を達成し続けること
サポート 問い合わせ・問題解決 リアクティブ（依頼を受けて動く） 問題が解消されること
営業 新規顧客獲得・売上拡大 商談・提案 契約・受注
PM（プロジェクトマネージャー） プロジェクトの完遂 納期・品質・リソース管理 期限内に成果物を納品すること

第二新卒がToB CSで最初につまずく3つのパターン

ACTION 1-1 — 役割の誤解
状況: 入社して最初の顧客担当になった。担当業務は「採用BPOの運用」。何をする人なのかを整理する。
❌ よくある失敗 「スカウトを送って、日程を調整して、報告すればいいんですよね」
→ 作業リスト思考。タスクを消化することを役割と誤解している。
✅ 正しいアプローチ 「お客様が採用目標を達成できるよう、スカウト・日程・報告を通じて採用プロセスが止まらない状態を維持するのが自分の役割」
→ アウトカム起点。最初にここを誤解すると、全ての行動がアウトプット視点になる。
ACTION 1-2 — つまずきパターン①：スカウト数は達成したのにクレームが来た
状況: 月間スカウト送付500件の目標を達成。しかし顧客から「採用できていない」と連絡が来た。
❌ よくある失敗 「契約通りスカウトは500件送りました。採用できるかどうかは候補者次第です」と返す
✅ 正しいアプローチ 「送付数は達成しましたが、通過率が先月より5%下がっています。ターゲット要件を見直しませんか」と提案する
POINT — スカウト送付はアウトプット。顧客が求めているのは「採用決定」というアウトカム。数字を達成しても顧客のアウトカムに近づいていなければ意味がない。
ACTION 1-3 — つまずきパターン②：日程調整の遅れで候補者が離脱した
状況: 候補者への面接日程調整を担当。返信が遅れていたが「まだ大丈夫だろう」と放置した結果、候補者が辞退。
❌ よくある失敗 辞退連絡を顧客に報告するだけ。「今回はご縁がなかったです」と説明する
✅ 正しいアプローチ 「日程調整の初回連絡から48時間以上空いていました。今後は24時間以内に初回連絡するルールを設けます」と再発防止策をセットで報告する
POINT — 問題発生後の報告だけでは信頼を失う。何が原因でアウトカム（内定承諾）が損なわれたかを示し、次の打ち手を提示することがCSの仕事。
ACTION 1-4 — つまずきパターン③：定例で何を報告すればいいかわからない
状況: 初めての顧客定例MTG。前任者のフォーマットを引き継いだが、数字を読み上げるだけになってしまった。
❌ よくある失敗 「今月のスカウト送付数は423件、書類通過は38名、面接設定は22名です」（数字の羅列）
✅ 正しいアプローチ 「今月は書類通過率が先月比3%下がりました。原因はターゲット職種のスカウト返信率低下と見ています。来月はメッセージ文面を改善するので、数字が戻るか一緒に見ていただけますか」
POINT — 数字を読み上げることはアウトプット。「何が起きていて、どうしたいか」まで伝えることがアウトカム視点の報告。

Chapter 1 まとめチェックリスト

  • CS担当者の役割を「顧客の目標達成を支援すること」と理解している
  • アウトプット（作業量）とアウトカム（成果）の違いを説明できる
  • 定例報告で「数字の変化・原因・次のアクション」をセットで伝えられる
  • 問題が起きたとき、再発防止策をセットで報告できる

Chapter 2: アウトカムの重要性

ゴール: 「作業を頑張る」から「顧客が何を得るか」に思考を切り替える
スコープ注記 この章で扱う「アウトカム視点」は、コンサル・PL・PM的な戦略提案は対象外。オペレーション業務の中で「何を届けているか」を明確にする範囲に限定する。

アウトプットとアウトカムの違い

種別採用BPOの例特徴
アウトプット（作業） スカウト送付100件 / 日程調整完了 / 報告書提出 作業の完了を示す。量で測れる。
アウトカム（成果） 採用決定2名 / 選考が滞りなく進んでいる / 担当者が状況を把握している 顧客が得た価値を示す。目標との差で測れる。

顧客は「作業量」ではなく「業務が回った結果」にお金を払っています。アウトカムを語れないCSは「ただの作業者」として扱われます。

ACTION 2-1 — アウトカムとアウトプットを区別する
状況: 月次報告書を作成する。今月は数字的に特段の変化はなかった。
❌ よくある失敗 「スカウト送付: 510件 / 書類通過: 42名 / 面接設定: 25名 / 内定: 3名」と表に並べる
✅ 正しいアプローチ 「内定3名（目標5名に対し進捗60%）。不足の主因は面接→内定フェーズの通過率低下（先月比-8%）。面接フィードバックを確認のうえ、来月中に要因分析をお届けします」
POINT — 同じ数字でも、目標との差・原因・次のアクションを加えると「アウトカムへの道筋」が見える報告になる。

顧客のアウトカムを問いかけで把握する

ACTION 2-2 — 顧客のアウトカムを問いかけで把握する
状況: 新規顧客のオンボーディング初日。まず「何を達成したいか」を確認する。
❌ よくある失敗 「今月のスカウト送付希望件数と対象職種を教えてください」（作業の入力収集）
✅ 正しいアプローチ 「今期の採用目標は何名で、いつまでに達成したいですか？今一番詰まっているフェーズはどこですか？」（ゴールと課題を先に把握）
採用BPO向け — アウトカム把握の質問例セット
  • 「今期の採用目標は何名ですか？どの職種が優先度高いですか？」
  • 「現在の選考通過率（書類通過・面接通過）はどのくらいですか？」
  • 「今一番詰まっているフェーズはどこですか？（応募獲得・書類選考・面接・内定後）」
  • 「前回の採用活動で、うまくいかなかった原因は何だと思いますか？」

追うべき数値を選ぶ（オペレーション範囲内）

ACTION 2-3 — 追うべき数値を選ぶ
❌ よくある失敗 「送付件数・通過件数・面接設定件数」だけ追う（工程の数量のみ）
✅ 正しいアプローチ 「送付件数・返信率・書類通過率・日程調整完了率・内定承諾率」を追い、各フェーズのボトルネックを可視化する
フェーズ追うべき指標（率）なぜ重要か
スカウト送付件数・返信率ターゲット精度・メッセージの質を示す
書類選考書類通過率要件設定の適切さを示す
日程調整日程調整完了率・所要時間対応スピードと候補者離脱リスクを示す
面接〜内定面接通過率・内定承諾率選考の精度と候補者体験を示す

数字を使った定例報告の組み立て方

ACTION 2-4 — 数字を使った定例報告の組み立て
状況: 今月、日程調整完了率が先月より10%下がった。定例で報告する。
❌ よくある失敗 「日程調整完了率が先月の82%から72%に下がりました」で終わる
✅ 正しいアプローチ 「日程調整完了率が72%（先月比-10%）です。候補者からの返信に平均で48時間以上かかっているケースが増えており、日程提示の幅が狭いことが原因と見ています。御社から候補者への第一連絡を3日以内にしていただけると、完了率が回復する見込みです」
POINT — 数字の変化→原因の仮説→改善のアクション（顧客側・自社側それぞれ）をセットで伝える。
定例報告テンプレート（数字を使ったアウトカム報告）

①今月の結果: [数字]（目標[数字]に対し[進捗%]）

②変化・異常: [指標]が[先月比+/-X%]（[良い/悪い]方向）

③原因の仮説: [具体的な原因]と見ています

④来月のアクション: 弊社側: [具体的アクション] / 御社側: [確認・依頼事項]

⑤確認事項: [1〜2点に絞った質問]

Chapter 2 まとめチェックリスト

  • アウトプット（作業量）とアウトカム（成果）を区別して報告できる
  • 初回ヒアリングで顧客の採用目標・課題フェーズを確認できる
  • 数量だけでなく「率」で各フェーズのボトルネックを可視化できる
  • 報告は「変化→原因→アクション」の順で組み立てられる

Chapter 3: スコープの話

ゴール: 「何をやる・やらない」を最初に握ることで、後の混乱を防ぐ
スコープとは: アウトカム（何を達成するか）の裏側にある「何をやらないか」がスコープです。スコープは「やること」だけでなく「やらないこと」まで言語化して合意することで機能します。言わなかった範囲は「やってくれると思っていた」になります。

スコープは契約範囲 × 期待値の管理で成り立ちます。どれだけ丁寧に仕事をしても、スコープが曖昧だと「もっとやってくれると思った」「なんでやってくれないの？」になります。

ACTION 3-1 — スコープを最初に握る
状況: キックオフMTGで担当範囲を確認する。
❌ よくある失敗 「では弊社でスカウト送付と日程調整を担当します。よろしくお願いします」と宣言して終わり
✅ 正しいアプローチ 「弊社の担当範囲はスカウト原稿作成・送付・日程調整・進捗報告です。面接の合否判定・内定条件の交渉はお客様側でお願いします。このご認識で合っていますか？」と境界を双方で確認する
キックオフで確認するスコープチェックリスト
  • 弊社が担当する業務の一覧（具体的なタスクレベルで）
  • 顧客側が担当する業務の一覧
  • グレーゾーン（どちらでもできる業務）の担当決定
  • 緊急時の連絡フロー・エスカレーション先
  • 報告の頻度・フォーマット・ツール
  • 追加依頼が発生した場合のルール（誰が・どうやって・いつまでに判断するか）
ACTION 3-2 — スコープ外依頼への断り方
状況: 担当者から「採用要件の設計も手伝ってほしい」と依頼が来た。契約範囲外。
❌ よくある失敗 「それは弊社の契約範囲外なので対応できません」と即答する
→ 即断りは関係を悪化させる。顧客は「困っているから頼んだのに」と感じる。
✅ 正しいアプローチ 「採用要件の設計は今の契約外になりますが、現在の要件でスカウト返信率が低い傾向があります。まず弊社で返信率データをまとめてお届けするので、それをもとに要件を見直すかどうかご検討いただけますか」
POINT — 「できません」だけでは関係が悪化する。契約内でできることで顧客の課題に近づく代替案をセットで返す。
スコープ外依頼への言い換えパターン集
❌ やってしまいがちな返し✅ 代替案をセットにした返し
「それは弊社の範囲外です」 「今の契約外ですが、〇〇で代替できます。いかがですか？」
「できません」 「今すぐは難しいですが、〇〇という方法ならできます」
「担当外なので他の方に聞いてください」 「正確にはXXが担当ですが、まず私から確認して折り返します」
「そこまではやっていません」 「そちらは追加の対応になりますが、正式にご依頼いただく形は可能ですか？」
ACTION 3-3 — 期待値コントロール
状況: 顧客から「来月中に5名採用できますか?」と聞かれた。現在の進捗では難しい。
❌ よくある失敗 「頑張ってみます」「できる限り対応します」と曖昧に答える
→ 曖昧な返答は「やってくれると思っていた」を生む。
✅ 正しいアプローチ 「現在の書類通過率と面接設定ペースだと、来月中に内定まで至るのは2〜3名が現実的な見込みです。5名を目指すには、スカウト送付数を今月比1.5倍にするか、ターゲット要件を緩和する必要があります。どちらを優先しますか？」
POINT — 数字ベースで現実の見通しを伝え、顧客に選択肢を渡す。「頑張ります」は期待値をコントロールできない最悪の返答。
ACTION 3-4 — スコープが広がるサインの早期検知
状況: 定例MTGで担当者から「ちょっとこれも見てもらえる?」が続いている。
❌ よくある失敗 「もちろんです！」と毎回引き受け、気づいたら作業が2倍になっている
→ 「ちょっと」が積み重なるとスコープが無限に広がる。
✅ 正しいアプローチ 「今回のご依頼は弊社の標準スコープ外になりますが、とても重要な課題だと思います。追加対応として正式にご依頼いただく形にすることは可能ですか？」と早期に線引きする
POINT — 親切心で引き受け続けると、CS担当者自身が疲弊し品質も落ちる。スコープ拡大は「最初の1回」で線引きするのが最も効果的。
スコープ拡大のサインリスト
  • 定例で「ちょっとこれも」が2回以上続く
  • Slackで追加依頼が頻繁に来るようになった
  • 担当業務の時間が増えているのに、報告内容が変わっていない
  • 「前回もこれやってくれたじゃないですか」と言われる

Chapter 3 まとめチェックリスト

  • キックオフでスコープ（やること・やらないこと）を双方で確認・合意できる
  • スコープ外依頼に対して、代替案をセットで返答できる
  • 期待値を問われたとき、数字ベースで現実的な見通しを伝えられる
  • スコープ拡大のサインに気づき、早期に線引きできる

Chapter 4: ToB担当者の解像度リアル

ゴール: 「担当者はわかってくれている」という前提を捨て、正しい伝え方を身につける
「ちゃんと理解してくれている」は幻想です。担当者はサービスについて思っているより解像度が低く、報告書は斜め読みしかしません。この前提に立つことで、伝え方が根本から変わります。

4-1: 担当者の解像度は思っているより低い

ACTION 4-1 — 担当者の理解度を確認する
状況: スカウト原稿を作成するため、ターゲット要件を担当者にヒアリングする。
❌ よくある失敗 「どんな人材を採用したいですか？」と聞いて、「営業経験3年以上で主体性のある人」という回答をそのままターゲットにする
✅ 正しいアプローチ 「営業経験3年以上とのことですが、現在活躍しているメンバーのご経歴を1〜2名教えていただけますか？また『主体性のある人』とは、具体的にどんな行動をしている人のことですか？」と深掘りする
POINT — 担当者が最初に言う要件は、実際に必要な要件の表面だけ。具体例を聞くことで本当のターゲット像が見えてくる。
ACTION 4-2 — 「わかった」を信用しない
状況: スカウト原稿の確認をお願いした。「いいですよ」と返ってきたが、送付後に「これは違う」と言われた。
❌ よくある失敗 「確認いただけたと思っていました…」と言い訳する
✅ 正しいアプローチ 次回からは原稿確認の際、「ここが意図と違う場合はNG」という具体チェック項目を添えて送る。承認後の変更はスケジュールに影響する旨を最初に伝える。
POINT — 「確認OK」は「全部読んだOK」ではない。確認してほしいポイントを明示し、齟齬が出たときのルールを事前に決めておく。

4-2: 報告書・提案書は斜め読みされる

ACTION 4-3 — 報告書は斜め読みされる前提で作る
状況: 月次レポートを作成する。今月は採用数が目標未達。
❌ よくある失敗 経緯・数字・分析・原因・対策を順番に書いた5ページのレポートを送る
→ 担当者は最初の1ページ目しか読まない。
✅ 正しいアプローチ 1枚目に「今月の結果: 内定2名（目標5名）/ 主因: 面接通過率の低下 / 来月の対策: 面接フィードバックシートの導入」をまとめ、詳細は2枚目以降に添付する
POINT — 担当者が最初に見るのは1枚目の上半分。「何が起きて、次どうするか」がそこで読み取れないと、詳細は読まれない。
報告書の構成 — Before / After
❌ 読まれない構成✅ 読まれる構成
1ページ目: 今月の経緯説明
2ページ目: 数字一覧
3ページ目: 原因分析
4ページ目: 来月の対策		 1枚目（要旨）: 結果・主因・対策を3行で
2枚目以降: 数字の詳細・分析・補足資料

4-3: 丁寧に伝える技術（対面・テキスト両対応）

ACTION 4-5 — 専門用語を使わずに伝える
状況: スカウト返信率が下がっていることを担当者に説明する。
❌ よくある失敗 「今月のレスポンスレートが6.2%で、先月比でCVRが低下しています」
→ 担当者はBPO業務指標の用語に慣れていない。
✅ 正しいアプローチ 「今月は100件スカウトを送って6人しか返信をいただけていません。先月は9人でした。メッセージの書き方を変えると改善できる可能性があります」
POINT — 数字を実感できる表現に変えると、担当者が課題を自分事として捉えやすくなる。
専門用語の言い換えコツ集
使いがちな用語担当者に伝わる言い換え
レスポンスレート / CVR「100件送って返信がきた数」「応募した人のうち次に進んだ割合」
スカウト返信率 6.2%「100件送って6人から返信が来た」
面接通過率が低下「面接を受けた方のうち次の選考に進む方が減っている」
オンボーディングが未完了「まだ運用の立ち上げが終わっていない」
チャーンリスクがある「解約につながる可能性があるサインが出ている」

テキストコミュニケーションの実務

ACTION 4-4 — テキストで催促する（返信がない場合）
状況: 候補者への面接案内を先方担当者に依頼して3日経つが返信がない。
❌ よくある失敗 「先日お送りした件、いかがでしょうか」と同じ文面で再送する
→ 何を確認してほしいか・なぜ今日必要かが伝わらない。
✅ 正しいアプローチ 「○○様、先週ご連絡した△△さんの面接日程の件です。候補者の方には今週中にご連絡したい旨お伝えしており、本日中にご確認いただけますと助かります。ご都合はいかがでしょうか？」
POINT — 返信がない理由は「忘れている」「忙しい」がほとんど。催促するときは「なぜ今日までに必要か」の理由と、「何を確認してほしいか」を1行で添える。
追いかけの変え方 — 1回目 / 2回目 / 3回目
回数トーン・内容文例
1回目
（3日後）		 軽めに、用件を明確に 「○○の件、ご確認いただけましたでしょうか。候補者への連絡が今週中なので、本日中にお返事いただけますと助かります。」
2回目
（さらに2日後）		 影響を具体的に 「度々失礼します。○○の件ですが、今週中のご連絡が難しくなってきました。候補者が他社に流れる可能性があるため、本日中にご判断いただけますか。」
3回目
（翌日以降）		 電話 or 別の手段に切り替え メッセージではなく電話で確認。その後「お電話しました。○○の件、明日の○時までにご確認いただけますか」とメッセージを残す。
口頭 vs 書面の使い分け判断基準
状況推奨手段理由
緊急・即判断が必要電話 → メッセージで記録スピードと記録を両立
複雑な内容・確認事項多いメール / ドキュメントで事前送付 → 口頭確認読み込んでから話すほうが齟齬が少ない
軽い連絡・進捗報告Slack / チャット往復コストが低い
合意・スコープの確定書面（メール / 議事録）後から「言った言わない」を防ぐ
クレーム・不満の受け止め電話 or 対面 → 後でメールで整理テキストは感情が伝わらず悪化しやすい

4-4: （応用）サービス理解を中長期で育てていく 応用編

ACTION 4-6 — 担当者のサービス理解を育てる（応用）
状況: 定例で毎回同じ質問をされる。「スカウト返信率ってどのくらいが普通なの?」
❌ よくある失敗 毎回「業界平均は5〜10%です」と答えるだけ
✅ 正しいアプローチ 「業界平均は5〜10%ですが、御社は現在6%です。弊社の他の支援先と比較すると、職種・メッセージ内容・送付時間帯の3つが主な変動要因です。参考として御社の過去3ヶ月のデータをグラフでお見せします」と可視化して渡す
POINT — 同じ質問への答えを毎回「提供」するのではなく、担当者が「自分で判断できる状態」に少しずつ近づけることが、長期的な信頼につながる。
「勉強させる」のではなく「体験させる」: 担当者に資料を読ませようとしても難しい。定例で一緒にデータを見ながら「なぜこうなっているか」を会話する方が自然に理解が深まります。

Chapter 4 まとめチェックリスト

  • ヒアリングで「具体例を聞く」習慣がある（抽象的な回答で終わらせない）
  • 「確認OK」を鵜呑みにせず、チェックポイントを明示できる
  • 報告書の冒頭1枚で「結果・主因・対策」を伝えられる
  • 専門用語を使わず、実感できる数字で伝えられる
  • 催促メッセージに「理由」と「期限」を添えられる
  • 口頭と書面を状況に応じて使い分けられる

Chapter 5: CSのスタンスと社内連携

ゴール: 担当者の解像度リアルを踏まえた上で、CSとして何をどうするかを行動レベルで決める

伴走者として動くとは

やること（伴走者）やらないこと（御用聞き）
問題が起きる前に提案する問題が起きてから報告する
指標を定点観測して変化に気づく担当者からの連絡を待つ
次のアクションをセットで提示する事実だけを伝えて終わる
顧客の状況を先読みして動く言われたことだけをこなす
ACTION 5-1 — 先回り提案をする
状況: 来月、顧客企業の新卒入社が重なりIT担当者が不在になることが分かった。日程調整に影響が出そう。
❌ よくある失敗 問題が起きてから「先方のご担当者が不在で日程調整が遅延しています」と報告
✅ 正しいアプローチ 2週間前に「来月の○週は担当者様が不在とのこと、その期間の日程調整は△△様に代行いただくか、事前に候補日を複数確保する形にしませんか」と提案する
POINT — 「起きてから報告」は信頼を損なう。「起きる前に提案」がCSの価値。担当者のカレンダーやイベントを把握しておくのはそのため。

「正直な情報提供」の実務

ACTION 5-2 — 悪いニュースを伝える
状況: 今月の内定者が0名になりそう。定例で報告する。
❌ よくある失敗 「今月は内定が出ませんでした。来月頑張ります」
→ 謝罪から入ると信頼が回復しにくい。
✅ 正しいアプローチ（事実→影響→対策の順） 「今月の内定者数は現時点で0名です（事実）。面接設定まで進んだ3名全員が辞退しており、日程提示のリードタイムが長かったことが主因です（影響と原因）。来月は48時間以内の初回連絡を徹底します（対策）」
悪いニュースを伝えるテンプレート

①事実: 「[具体的な数字・状況]です」（言い訳なし）

②影響と原因: 「[何に影響していて]、[何が原因]と見ています」

③対策: 「[具体的なアクション]を[いつまでに]実施します」

社内ステークホルダーとの連携アクション

ACTION 5-3 — 社内の営業に情報を渡す
状況: 顧客が「来期は採用ポジションを3つに増やしたい」と言っていた。
❌ よくある失敗 「そういえば言ってましたよ」と口頭で営業に伝えるだけ
✅ 正しいアプローチ 「○月○日の定例で、来期3ポジション追加の話が出ました。現在の契約範囲では対応できない可能性があります。営業から先方へのヒアリングを来月末までにお願いできますか？」とメモで共有し期日を添える
POINT — CS担当者が持っている顧客情報は、営業にとって最も価値のあるインテリジェンス。口頭で流すだけでなく記録に残し、アクションを促す。
連携先何を・いつ・どの形式で渡すか
営業 顧客の拡大意向・不満シグナル・解約リスクを定例後3日以内に書面で共有。アクション依頼も添える。
開発・PM 「顧客が困っている」だけでなく「どういう対処が欲しい」まで変換して伝える。感情的な要望は事実ベースに翻訳する。
責任分界 CS=顧客成功支援・継続運用 / 営業=契約・拡大 / PM=プロジェクト完遂。境界が曖昧なときは担当者に聞く前に社内で確認する。
ACTION 5-4 — ツールに記録を残す
状況: 顧客からSlackで「次回の定例は延期したい」と来た。
❌ よくある失敗 「了解です」とSlackで返し、メモはしない
→ 担当者が変わったとき「聞いていた話と違う」が起きる。
✅ 正しいアプローチ 「了解です」と返しつつ、CRM/タスク管理ツールに「○月定例: ○日に延期（先方都合）」と記録。次回定例のアジェンダも更新する
POINT — CSの仕事は担当者が変わっても継続する。記録がなければ「前の担当者はやってくれていた」が起きる。
最低限の記録ルール
  • 顧客からの依頼・合意・変更は必ずツール（CRM・タスク管理）に記録する
  • 口頭で決まったことは「先ほどの件、〇〇で合意しました」とメッセージで確認を取る
  • 定例は毎回アジェンダと議事録を残す（簡易版でよい）
  • 「言った言わない」が起きそうなことはメール/書面で確認を取る

Chapter 6: 場面別アクション集

ゴール: 実務でよく遭遇する場面で、迷わず動けるようにする

場面①: 採用BPOオンボーディング（最初の30日）

ACTION 6-1 — 最初の30日の動き方
❌ よくある失敗 キックオフ後すぐにスカウト送付を開始。ターゲット要件は大まかに確認しただけ。定例は翌月から。
→ 走り方を合わせる前に走り始めると、後で大きく修正が必要になる。
✅ 正しいアプローチ 速く走り始めるより「走り方を合わせること」が先。最初の30日で体制の合意を取りきる。
最初の30日チェックリスト（採用BPO）
時期やること目的
Week 1 スカウトターゲット要件の詳細ヒアリング（活躍社員の事例を3名分収集） ターゲット像を具体化する

場面②: 顧客定例のファシリテーション

ACTION 6-2 — 顧客定例の進め方
❌ よくある失敗 数字を読み上げる → 「何かご質問はありますか？」 → 終了。毎回同じ流れ。
→ 顧客が定例後に「何をすればいいか」が明確にならない。
✅ 定例の進め方（推奨4ステップ）
  1. 先月の振り返り（数字 + 1つの発見）
  2. 今月の焦点（どのフェーズを改善するか）
  3. 先方への確認事項（1〜2点に絞る）
  4. 次回までのアクション確認（CS側・顧客側それぞれ）
顧客定例アジェンダテンプレート（採用BPO / 30分）
時間項目内容
5分先月の振り返りスカウト送付数・返信率・書類通過率・日程調整完了率・内定数（目標比）
5分今月の1つの発見「今月は○○が改善/悪化しており、理由は○○と見ています」
10分今月の焦点改善すべきフェーズと取るべきアクションを合意
5分先方への確認事項1〜2点に絞った確認（ターゲット要件変更・採用計画の変化等）
5分次回までのアクション確認弊社側: 〇〇 / 御社側: 〇〇（担当者名・期限を明示）

場面③: 担当者交代時の引き継ぎ

ACTION 6-3 — 担当者交代時の引き継ぎ
❌ よくある失敗 旧担当者に「新しい方によろしく伝えておいてください」とお願いするだけ
→ 「聞いていなかった」が必ず起きる。
✅ 正しいアプローチ ①引き継ぎ資料を自社で作成（現在の進捗・スコープ・ルール・過去の経緯） → ②新旧担当者＋CSの三者で30分の引き継ぎMTGを設定 → ③新担当者から「理解できた」の確認を書面でもらう
POINT — 引き継ぎの主体はCS担当者。顧客側に任せると「聞いていなかった」が必ず起きる。
引き継ぎ資料に含めること
  • 現在の採用目標と進捗状況（直近3ヶ月の数字）
  • 担当範囲（スコープ）と合意している運用ルール
  • これまでの主なトラブルと対処経緯
  • 定例の頻度・フォーマット・使用ツール
  • 顧客のこだわりポイント・地雷（過去のフィードバックから）

場面④: クレーム・不満を受け止める

ACTION 6-4 — クレーム・不満の受け止め方
状況: 担当者から「なんか最近対応が遅い気がする」とSlackが来た。
❌ よくある失敗 「対応時間は契約通りです。SLAに違反していないか確認します」と返す
→ 契約を盾にすると関係が悪化する。
✅ 正しいアプローチ（受け止め → 事実確認 → 改善策） 「ご不便をおかけして申し訳ありません。具体的にどの対応が遅いとお感じでしたか？事実を確認したうえで、改善策をお伝えします」と聞く
POINT — 最初の返答は「受け止め」が最優先。事実確認・弁解は後。顧客が「ちゃんと聞いてもらえた」と感じることが、怒りを収める最初のステップ。

場面⑤: 解約リスクを感じたときのアクション

ACTION 6-5 — 解約リスクの早期検知と対応
状況: 最近、定例の欠席が増え、レスポンスも遅くなってきた。担当者の反応も薄い。
❌ よくある失敗 「最近忙しいのかな」と思いながら様子を見る
→ 解約の意思決定は静かに進む。「様子を見る」は最悪の対応。
✅ 正しいアプローチ（シグナル検知 → 確認 → 改善提案） ①シグナルを記録（欠席回数・レスポンス時間の変化） → ②「最近ご状況はいかがでしょうか」と個別で連絡し温度感を確認 → ③「現在の支援で課題に感じている点はありますか？」と率直に聞く → ④問題があれば改善提案を1週間以内に出す
解約リスクのサインリスト
  • 定例の欠席・遅刻が増えた（2回連続は要注意）
  • レスポンス時間が以前より明らかに遅くなった
  • 定例での発言量が減り、「わかりました」だけで終わるようになった
  • 担当者の役職・部署が変わった（方針変更の可能性）
  • 「他にもいいサービスあるか調べてる」という言葉が出た
