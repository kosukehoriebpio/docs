1. システム構成（3層アーキテクチャ）

📝

Notion

業務オペレーションUI

  • 顧客DB（既存）
  • 議事録DB
  • マニュアルDB
  • タスクDB
  • ナレッジDB

  • 全データの正規化DB
  • SQL分析・集計
  • BIツール接続用
  • API提供
  • リアルタイム同期

利用者: システム（裏方）

⇄
📊

HubSpot

マーケティング / CRM

  • リード管理
  • パイプライン管理
  • フォーム・LP作成
  • メール配信
  • 広告管理

利用者: マーケ担当 1-3名

←
📈

Looker Studio

BIダッシュボード

  • ファネル分析
  • チャネル別ROI
  • MRR/LTV推移
  • 解約率分析
  • GA4連携

利用者: 経営・マーケ担当

Notionは「業務UI」、Supabaseは「データ基盤」、HubSpotは「マーケツール」、Looker Studioは「可視化」
それぞれ得意な役割に集中させ、Supabaseをハブとして同期する

2. データ同期フロー

1
Notion → Supabase
定期同期（10-30分ごと）：顧客情報・契約ステータス・報酬データをNotionから取得しSupabaseに反映
2
HubSpot → Supabase
Webhook（リアルタイム）：リード登録・ステータス変更・メール開封などのイベントをSupabaseに記録
3
Supabase → Notion
イベント駆動：HubSpotで成約 → Supabase更新 → Notion顧客DBに新規クライアントとして自動追加
4
Supabase → HubSpot
必要時のみ：既存顧客情報をHubSpotに渡す（アップセル管理・既存顧客のリエンゲージメント）
5
Supabase → Looker Studio
PostgreSQL直接接続：リアルタイムでダッシュボードに反映。SQLクエリで自由にKPIを算出

7. 算出可能なKPI一覧

ファネル系KPI

チャネル別リード数
SELECT channel, COUNT(*) FROM leads GROUP BY channel
→ どのチャネルに投資すべきか判断
チャネル別商談化率
商談化リード数 / 全リード数 × 100 GROUP BY channel
→ チャネルの「質」を評価
チャネル別成約率
成約deals / 全deals GROUP BY source_channel
→ 最終的にどのチャネルが成果を出しているか
チャネル別CPA（リード獲得単価）
channels.monthly_cost / リード数
→ コスト効率の比較
チャネル別CAC（顧客獲得コスト）
channels.monthly_cost / 成約数
→ 1顧客を獲得するのにいくらかかるか
平均リードタイム
AVG(deals.actual_close_date - leads.first_touch_at)
→ 営業プロセスの速度評価

収益系KPI

MRR（月次経常収益）
SUM(contracts.monthly_revenue) WHERE status='継続中'
→ 事業の健全性指標
サービス別MRR
SUM(monthly_revenue) GROUP BY service_type
→ サービスポートフォリオの評価
平均契約単価
AVG(contracts.monthly_revenue)
→ 価格戦略の検討材料
LTV（顧客生涯価値）
平均月額 × 平均契約月数
→ チャネル別ROIの最終判断に使用
LTV/CAC比率
LTV / CAC（3以上が理想）
→ マーケティング投資の回収効率
Net Revenue Retention
前月MRR + アップセル - ダウングレード - 解約
→ 既存顧客からの成長力（100%超が理想）

顧客健全性KPI

月次解約率（Churn Rate）
当月解約数 / 前月末アクティブ顧客数 × 100
→ 顧客維持の健全性指標
解約理由分布
SELECT churn_reason, COUNT(*) FROM accounts WHERE churned_at IS NOT NULL GROUP BY churn_reason
→ 解約防止策の優先順位付け
規模別・業種別の成約率
成約数 / リード数 GROUP BY revenue_range, industry
→ ターゲティング精度の検証・改善
